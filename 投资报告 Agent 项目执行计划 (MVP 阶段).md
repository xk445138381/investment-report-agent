# 投资报告 Agent 项目执行计划 (MVP 阶段)

## 1. 项目概述

本项目旨在开发一个基于多 Agent 协作的投资报告生成系统，通过集成金融数据、深度分析模型和多空辩论机制，为专业机构提供高质量的个股深度研究报告。本计划涵盖了从需求完善到 MVP 交付的 6 周执行周期。

## 2. 阶段划分与里程碑

| 阶段 | 周期 | 核心目标 | 里程碑 |
| :--- | :--- | :--- | :--- |
| **Phase 1: 需求完善与架构定型** | 第 1 周 | 完善需求文档，确定技术规格与接口协议 | 需求文档 V1.1 发布，接口协议定稿 |
| **Phase 2: 数据层与计算层开发** | 第 2 周 | 建立数据管道，实现核心金融计算逻辑 | 数据 Provider 联通，计算模块通过校验 |
| **Phase 3: Agent 编排与 Prompt 工程** | 第 3-4 周 | 开发各 SubAgents，完成编排逻辑与 Prompt 调优 | 核心 Pipeline 跑通，辩论逻辑闭环 |
| **Phase 4: 模板系统与 UI/UX 实现** | 第 5 周 | 实现报告渲染引擎，完成前端核心交互界面 | 报告 PDF 成功渲染，Web 预览功能上线 |
| **Phase 5: 测试、合规与 MVP 交付** | 第 6 周 | 完成全链路测试，确保合规性，正式交付 MVP | MVP 版本发布，通过验收标准 |

## 3. 详细任务分解

### Phase 1: 需求完善与架构定型 (Week 1)
- **文档补全**：
    - 明确 15 项核心决策的优先级与依赖关系。
    - 补充决策风险评估及缓解策略。
    - 增加项目 SWOT 分析及商业竞品对比。
- **技术规格定稿**：
    - 编制详细数据字典（字段、来源、频率）。
    - 定义 RESTful API 接口规范（OpenAPI 规范）。
    - 绘制 Agent 编排详细流程图。
- **合规与法律**：
    - 寻求专业法律咨询，确认数据使用合规性。
    - 完善 ToS 条款及隐私政策说明。

### Phase 2: 数据层与计算层开发 (Week 2)
- **数据管道建设**：
    - 实现 AkShare/Tushare/Yahoo Finance 数据 Provider。
    - 建立多级缓存机制（Memory + SQLite）。
    - 实现数据质量校验规则及异常处理流程。
- **计算层实现**：
    - 封装技术指标、财务比率、估值模型计算模块。
    - 实现计算参数配置化（WACC、增长假设等）。
    - 进行计算精度校验与性能初步优化。

### Phase 3: Agent 编排与 Prompt 工程 (Week 3-4)
- **SubAgent 开发**：
    - 开发数据聚合、财务分析、估值建模等核心 Agent。
    - 开发多空辩论 Agent 及风险裁判 Agent。
- **编排逻辑实现**：
    - 使用 LangGraph 构建 Orchestrator 调度逻辑。
    - 定义 Agent 间通信协议及状态管理。
- **Prompt 调优**：
    - 应用指令式、事实锚定等最佳实践进行 Prompt 迭代。
    - 建立 Prompt 版本管理机制。
    - 针对复杂场景进行 Agent 行为测试。

### Phase 4: 模板系统与 UI/UX 实现 (Week 5)
- **报告渲染引擎**：
    - 编写 Jinja2 报告模板，实现数据变量绑定。
    - 集成 WeasyPrint 实现 HTML 转 PDF/Word。
    - 实现图表自动生成与嵌入方案。
- **前端交互开发**：
    - 实现对话式交互界面及报告预览功能。
    - 开发基础模板管理界面。
    - 融入用户反馈机制（评分、修改建议）。

### Phase 5: 测试、合规与 MVP 交付 (Week 6)
- **全链路测试**：
    - 执行集成测试与端到端测试。
    - 进行性能压力测试及安全性扫描。
    - 细化测试用例管理，跟踪 Bug 修复。
- **交付准备**：
    - 按照验收标准进行 MVP 最终核对。
    - 准备部署文档及用户操作手册。
    - 正式发布 MVP 版本。

## 4. 资源与角色分配

-   **项目经理 (PM)**: 负责进度跟踪、风险管理及合规审查。
-   **后端开发 (BE)**: 负责数据层、计算层、API 及 Agent 编排实现。
-   **前端开发 (FE)**: 负责 UI/UX 交互、报告预览及模板系统前端。
-   **AI/Prompt 工程师**: 负责 Prompt 设计、调优及 Agent 行为测试。
-   **测试工程师 (QA)**: 负责全链路测试、数据质量校验及验收测试。

## 5. 风险管理

| 风险项 | 影响程度 | 缓解策略 |
| :--- | :--- | :--- |
| **数据源 API 限制/失效** | 高 | 建立多源备份机制，优化缓存策略，预留 Mock 数据模式。 |
| **Agent 输出质量不稳定** | 高 | 强化 Prompt 约束，引入人工审核环节，建立 A/B 测试机制。 |
| **计算逻辑复杂导致性能瓶颈** | 中 | 采用向量化计算，优化数据库查询，必要时引入异步处理。 |
| **合规性风险** | 极高 | 提前进行法律咨询，严格遵守数据提供商 TOS，明确免责声明。 |

## 6. 验收标准 (Success Metrics)

1.  **功能完整性**：成功生成包含 4-6 个核心章节的单标的深度报告。
2.  **数据准确性**：报告中的金融数据与原始数据源一致性达到 100%。
3.  **生成成功率**：在正常网络环境下，报告生成成功率不低于 95%。
4.  **用户满意度**：MVP 试用用户对报告质量的评分不低于 4/5 分。
5.  **性能指标**：单份深度报告的生成时间控制在 3-5 分钟以内。
