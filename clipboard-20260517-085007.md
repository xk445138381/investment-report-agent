下面是对整个项目（investment-report-agent）做的整体分析。我读了 README、`backend/`（config/orchestrator/api/providers/calculators/export/models/tests）、`backend/pyproject.toml`、`frontend/package.json` 以及部分前端代码与 e2e 测试，并实际跑了一遍后端测试。

# 一、整体观感
这是一个"门面完整、内核空心"的脚手架。

- 架构图、文档、配置 schema、Pydantic models、PDF/HTML 模板引擎、Provider 路由+熔断、估值/财务计算公式、orchestrator 调度框架、FastAPI 路由、Next.js 首页、Playwright e2e —— **骨架全部摆出来了**。
- 但是几乎所有真正"干活"的部分都是 stub：4 个 agent 子包（data/analysis/debate/assembly）里 `__init__.py` 全空；orchestrator 的 `_run_agent` 自己注释里就写着 "this is a stub that logs"；ProviderManager 没有任何 concrete provider 实现（AkShare/Yahoo 都不存在）；upload 路由读完文件直接丢弃；user 路由返回硬编码用户。
- README 自吹的"28 财务比率 + DCF + Owner Earnings + EV/EBITDA + AkShare/Yahoo F./PDF/Word + 81 unit + 7 E2E + LangChain/Claude/OpenAI/DeepSeek 可配置"中，**真正落地的只有 calculators 的纯 Python 公式和 template_engine 的一段 HTML 模板**，其余全部是占位。
# 二、严重问题（会让项目"跑不起来"或行为错误）
### 1. `pyproject.toml` 严重缺依赖
`pyproject.toml` 只声明了 `pydantic / python-jose / structlog / pyyaml`，但代码里实际 `import`：

- `fastapi`、`starlette`（api/main.py、所有 routes）
- `jinja2`（export/template_engine.py）
- `weasyprint`（export/pdf_renderer.py）
- `langchain_anthropic` / `langchain_openai` / `langchain_ollama`（config/loader.py）
- README 还提到 AkShare、yfinance，但 providers 里也没用上。
实测：`pip install -e ".[dev]"` 之后 `pytest` 直接报 `ModuleNotFoundError: No module named 'langchain_anthropic'`，`test_config.py::test_resolve_anthropic_model_requires_env_key` 失败。`uvicorn src.api.main:app` 在干净环境里也起不来（缺 fastapi）。

→ README 写的 "73 passed, 1 skipped" 我这边复现是 1 failed / 73 passed / 1 skipped（在装上 fastapi 后）。

### 2. `frontend/package.json` 版本号是未来版本
Code









```
"next": "16.2.6",
"react": "19.2.4",
"react-dom": "19.2.4",
"eslint-config-next": "16.2.6"

```


Next.js 当前 stable 是 15.x，没有 16.2.6；React 19.2.4 也不存在。`npm install` 在公网会失败 / 拉到错误版本。看起来是人工瞎填或被 AI 写错。

### 3. 估值公式有真正的数学错误
`calculators/valuation_engine.py`：

- **DCF 重复计入现金**：

Python









```
equity_value = enterprise_value - net_debt + cash - minority_interest

```


`net_debt = total_debt - cash`，已经把 cash 净掉了。再 `+ cash` 等于把现金算了两遍。正确写法应是 `EV - net_debt - minority_interest` 或 `EV - total_debt + cash - minority_interest`。
- **EV/EBITDA 少数股东权益符号反了**：

Python









```
equity_value = enterprise_value - net_debt + minority_interest + preferred_stock - non_controlling

```


标准公式应是 `EV - debt + cash - minority - preferred`。这里 minority 和 preferred 都被加号；non_controlling 又和 minority 是同义词被减一次，整体不自洽。
- **Owner Earnings 的"5 年预测增速"被写成了 `terminal_growth=3%`**，且对 `mos` 是先折现完再 `* (1 - mos)`，相当于直接在内在价值上再打 0.75 折——是否符合作者意图存疑（一般 Buffett 流派的安全边际是和市价比较时使用，不是直接乘进 PV）。
- **`calculate_dcf_three_stage` 没有用 WACC 输入参数 `beta / equity_weight / debt_weight / interest_expense / total_debt / tax_rate`**，函数签名收了一堆 WACC 组件，但内部只用了外部传入的 `wacc`，根本没算 CAPM/WACC。要么删掉这些参数，要么真的实现。
### 4. Orchestrator 的 task_id 类型不一致 + 路由逻辑没用
- `AgentContext.task_id: Optional[UUID]`，但 `routes/report.py` 传 `task_id=str(uuid4())`——字符串。Pydantic 类型契约违反（虽然 AgentContext 不是 BaseModel 没报错）。
- `routes/report.py:41` 调用 `orchestrator.route(req.ticker, {})` ——把 **股票代码**当成 **用户消息**喂给意图识别正则；并且返回的 `pipeline_type` 完全被丢弃，最终仍走 `req.report_type`。整个 `route()` 函数事实上是死代码。
- `route()` 的 US ticker 正则 `\b([A-Z]{1,5})\b` 在中文消息里会匹配到很多奇怪字符串（黑名单只有 `HK/SH/SZ/IPO`），健壮性很差。
- "看下" 同时命中 `brief` 关键字和中文公司名提取正则，可能产生路由冲突。
### 5. 进度回调是 fire-and-forget
`AgentContext.set_progress`:

Python









```
asyncio.ensure_future(self.progress_callback(phase, pct, message))

```


- 不 await，异常被吞；
- 在非 running loop 上下文（例如同步调用）会抛 `RuntimeError`；
- `_internal_pct` 每个 agent 自增 10，没有上限，可能超过 100。
SSE 流也只在 `current_phase != last_phase` 时推一帧，跨 phase 的细粒度进度看不到；不发送心跳，客户端/代理容易超时断开。

### 6. `_run_serial` 没用 `phase_cfg.debate_rounds`
Python









```
async def _run_serial(self, ctx, phase, phase_cfg, agent_names):
    for name in agent_names:
        await self._run_agent(ctx, name, phase)

```


辩论轮数完全没生效。schema 里 `debate_rounds` 字段形同虚设。

### 7. 全局可变状态、不可水平扩展
`_tasks` / `_uploads` / `_templates` 都是模块级 dict。多 worker 必然不一致；进程重启数据丢失；并发安全没保护；没有过期清理 → 内存泄漏。

`_orchestrator` 单例，第一次拿到的 config 之后永远不会更新；如果 `set_config` 更新了配置，orchestrator 不会感知。

# 三、安全 / 健壮性问题
### 8. 文件上传无任何防护
`routes/upload.py`：

- 无大小限制 → 大文件 OOM；
- 无 MIME / 扩展名校验；
- `await file.read()` 一次全读进内存；
- 内容立刻丢弃，只存 `len(content)`，所以上传功能实际上**根本没有保存文件**，后续 pipeline 也无从消费。
### 9. Jinja2 autoescape 实际未启用，存在 XSS 风险
Python









```
Environment(loader=BaseLoader(), autoescape=select_autoescape())

```


`select_autoescape()` 默认仅对 `.html/.htm` **模板文件名**启用 autoescape；这里模板是 `from_string` 的字符串，**没有文件名**，因此 autoescape **始终是 False**。`metadata.title / sections.content / chart.caption` 都未转义，如果数据来源不可信会导致 HTML 注入（生成 PDF/页面）。应改为 `autoescape=True` 或 `select_autoescape(default=True, default_for_string=True)`。

### 10. 鉴权完全没有
- `python-jose` 引入了但全项目没引用一行；
- `/user/me` 返回硬编码 `user@example.com`；
- 没有任何 `Depends(get_current_user)`；
- 配额 / 订阅 tier 文档里有，代码里没有。
### 11. CORS 只白名单 localhost:3000
生产部署需要可配置。

### 12. `provider_name` 没有强制
`DataProvider.provider_name` 在基类被声明但没有 `@property` / `@abstractmethod`，子类不实现也不会报错；`ProviderManager.__init__` 里 `provider.provider_name` 可能拿到 `None`。

### 13. config.json 使用了不存在的模型
`"model": "claude-opus-4-7"` 不是有效的 Anthropic 模型 id。

# 四、可维护性 / 质量问题
- **未使用的 import**：orchestrator.py 里 `from datetime import date`、`from typing import Callable, Awaitable`（OK 用了）, `Optional`（OK），`UUID`（用了）；不算多但 `date` 没用。
- **`AgentContext.created_at`**：写入未读。
- **`config_accessor.py`** 通过 `from config.loader import Config` 拿 Config，依赖 loader 把 schema.Config 顺带 re-export 的副作用，不太干净。
- **`api/routes/__init__.py`** 没有把 routes 显式暴露（虽然 main.py 直接 import 子模块也能用）。
- **日志没初始化**：所有模块 `logging.getLogger(__name__)`，但 `LoggingConfig` 里的 level/output/file_path 从来没人读。FastAPI 启动后默认还是 uvicorn 的 logger 配置。
- **README 文档与现实不符**：- 引用 `frontend/DESIGN.md` —— 仓库没有；
- 引用 `.context/saturday-may-16-foamy-shore.md` —— 仓库没有；
- "Python 计算层 28 财务比率" —— `financial_ratios.py` 实际函数数远不到 28；
- "数据 AkShare/Yahoo Finance/用户上传/本地缓存" —— 全部未实现，连 `import akshare` / `import yfinance` 都没有。
- **`frontend/test-results/.last-run.json` 被提交**——属于运行产物，应该在 `.gitignore` 里。
- **大量中文 markdown（需求文档/风险评估/任务列表）放在仓库根**，应统一归到 `docs/` 目录。
- **`backend/tests/integration/`** 是空的，README 却宣传"7 E2E"全在前端。
- **`pyproject.toml`** 没声明 package 路径（`[tool.setuptools] packages` 或 `src` 布局配置），`pip install -e .` 能跑是因为不需要 import 包，但发布会有问题。
- **`mypy strict=false`** + 没 CI —— 类型问题没法发现（`task_id: UUID` 接收 str 就是个例子）。
- **没有 `Dockerfile / docker-compose`**，部署文档为零。
- **没有 GitHub Actions / CI workflow**（`.github/workflows/` 缺失），README 写的测试命令谁都不会自动跑。
# 五、建议优先修复顺序
1. **修 `pyproject.toml`**：补 fastapi/uvicorn/jinja2/weasyprint/langchain-* 等依赖；把 `setuptools` 包结构写对；用 `pip-compile` 锁版本。
2. **修 `frontend/package.json`**：把 next/react 改回真实存在的稳定版本（next 15.x / react 19.0.0 等），重新 `npm install` 验证。
3. **修估值公式**：DCF 现金重复、EV/EBITDA 符号、Owner Earnings 增速三处数学错误，并补单元测试覆盖。
4. **修 Jinja2 autoescape**（一行改动，避免 XSS）。
5. **修 orchestrator**：task_id 类型统一、`route()` 真正被使用 / 或者从 `report.py` 删除；`set_progress` 改为 `await`；`_run_serial` 实现 `debate_rounds`。
6. **上传接口加入大小/类型限制并真正持久化**（或临时存到 cache_root）。
7. **加最小可用的认证**（哪怕是固定 token 中间件），把 `python-jose` 用起来或删掉。
8. **加 CI（GitHub Actions）**：跑 ruff、mypy、pytest、playwright；提交前能阻止上面这些问题再次进入。
9. **把 stub agent 真的填一个端到端最小实现**（哪怕 data agent 只能从用户上传 csv 读财务）—— 否则整个 pipeline 没有任何端到端价值。
10. **清理 README**：去掉尚未实现的能力，或者明确标注 "Planned"，避免读者被误导。
