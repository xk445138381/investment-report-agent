## 🔴 必须修复（功能性 bug）
### 1. `financial_analysis_agent._score` 的满分实际不可达
`backend/src/agents/analysis/financial_analysis_agent.py`

- `_to_income` 把 `cogs=None`、`interest_expense=None`，下游 `calculate_all_ratios` 会把 `gross_margin`、`interest_coverage` 全部算成 `None`；
- `calculate_all_ratios` 里 `revenue_growth_yoy` / `earnings_growth_yoy` 本就硬编码为 `None`（需要多年数据），但 `_score` 给 growth 留了 **8 分**；
- 同理 `interest_coverage` 留 3 分、`gross_margin` 留 3 分，全部拿不到。
结果：理论 32 分，实际天花板大约 **18 分**，永远拿不到 `EXCELLENT`（阈值 26），`GOOD` 也几乎够不着。评级毫无区分度。

建议：

- 要么把 financials 改为接收 list 然后在这里自己算 YoY（推荐，已经收到 `financials` 列表）；
- 要么按"可用维度归一化"，例如 `score / max_available * 100` 再分级；
- `cogs`、`interest_expense` 如果数据源真的没有，应在 `_score` 里把对应分项从分母里扣掉，而不是默默丢分。
### 2. `industry_agent.moat_assessment` 永远是 `NARROW`
Python









```
ratios = {"roe": None, "revenue": ..., "net_income": ...}
...
"overall_moat_width": "WIDE" if (ratios.get("roe") or 0) > 0.20 else "NARROW"

```


`ratios["roe"]` 写死成 `None`，护城河判断恒为 `NARROW`。要么真的算一次 ROE（`net_income / total_equity`），要么去掉这个伪判断。

### 3. `section_writer._fin_analysis` 把合法的 0 当成缺失
Python









```
content = f"ROE: {roe*100:.1f}% | 负债权益比: {de:.2f}" if roe and de else "财务指标待补充"

```


`debt_to_equity = 0`（无负债）是合法且非常正面的情况，但这里短路成"待补充"。应使用 `roe is not None and de is not None`。同样问题在 `_exec_summary` 里的 `'上行' if upside>0 else '下行'`——`upside == 0` 时会被误判为"下行"。

### 4. `section_writer._val_section` 数字格式化崩塌
Python









```
f"加权目标价: {v.get('weighted_value','N/A')}"

```


`weighted_value` 通常是浮点数（如 1834.572834）。当前会直接渲染成长尾小数；而当字段缺失时变成字符串 `"N/A"`。两者类型不同。建议统一：

Python









```
val = v.get('weighted_value')
val_str = f"{val:.2f}" if isinstance(val, (int, float)) else "N/A"

```


### 5. `chart_agent` 中文乱码（PDF 体验关键问题）
matplotlib 默认 font 不带 CJK，标题/坐标轴上的"股价走势""营收""净利"等会全部渲染成 ☐☐☐。这是导出 PDF 报告的硬伤。需要：

- 显式设置 `plt.rcParams["font.sans-serif"] = ["Noto Sans CJK SC", "SimHei", ...]`；
- 设置 `plt.rcParams["axes.unicode_minus"] = False`；
- 并在 `backend/pyproject.toml` 文档里写明系统需要安装中文字体（或随项目附带 .ttf）。
### 6. `section_writer` 评级文案漏掉 `SELL`
Python









```
f"{'买入' if 'BUY' in verdict else '持有'}评级"

```


当 judge 返回 `SELL` / `STRONG_SELL` 时也显示"持有"，会误导投资者，属于合规风险。建议显式映射 4 档（强买/买入/持有/卖出）。

## 🟡 建议改进（可读性 / 健壮性）
1. **`section_writer_agent.py`**：`import json` 没用到；从 `financial_data.result.ratios` 取值的代码路径其实拿不到（`financial_data` 不返回 `ratios`），靠 `fa.get("ratios", {})` 兜底。删掉死代码会更清晰。
2. **`financial_analysis_agent` 字典推导式**：

Python









```
{k: round(v, 4) if v else None for k, v in ratios.items() if v is not None}

```


`if v` 把 `0` 也当作 falsy，导致合法的 0（如 `interest_expense=0`、`debt_to_equity=0`）被替换成 `None`，与 `if v is not None` 过滤的语义矛盾。改为 `round(v, 4) if isinstance(v, (int, float)) else v`。
3. **`orchestrator._dispatch_agent`** 出现两段 `# Phase 2: Analysis agents` 注释；`financial_analysis` 被放在 `risk_judge` 之后，而它本该在 bull/bear 之前跑。如果 pipeline config 里依赖顺序错了，bull/bear 会拿到空的 `financial_analysis`。**强烈建议确认** `config.json` 里 `phase_2` 的 agent 顺序，以及在 phase 内是 sequential 还是 parallel 启动 bull/bear。这是潜在的"数据空指针式"风险。
4. **`_to_income.tax_rate=0.25`、`ebitda=operating_income`、`total_debt=total_liabilities`** 全是粗暴近似，至少加注释 `# TODO: refine when 折旧/摊销 fields available`，避免下游团队误以为是精算值。
5. **`chart_agent` 返回 dict 而不是 `ChartOutput` model**：与 `backend/src/models/` 下其它 agent 的风格不一致。如果下游（PDF 渲染）用 pydantic 校验会失败。请确认 export 层是否做了显式转换。
6. **`summary_agent.tags = [..., company_name[:4]]`**：截断"贵州茅台股份"得到"贵州茅台"还行，但截断"中国国际航空公司"得到"中国国际"语义不佳。可以考虑用 `company_name.split('股份')[0]` 或前 2 个字 + 简称表。
7. **`governance_agent` / `news_agent` / `macro_agent`** 是 6 行的"待补充"占位，建议至少：

- 在 result 里返回固定的 `"status": "stub"` 字段；
- orchestrator 里识别该 status 后跳过下游对它的依赖，避免污染最终 report。
8. **缺测试**：`backend/tests/` 已存在测试体系，但本次新增 8 个 agent 没有任何单测。最低限度建议加：

- `test_financial_analysis_agent.py`：mock 一份高 ROE、低杠杆 financials，断言 rating 不应永远是 POOR；
- `test_section_writer.py`：构造各种 verdict 验证文案；
- `test_chart_agent.py`：断言返回的 base64 能被 `base64.b64decode` 且 PNG header 正确。
